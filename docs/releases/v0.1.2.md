# Kybernetes v0.1.2

Date: 2026-07-17

Kybernetes v0.1.2 separates rejection-capable evidence from accountable
acceptance when work crosses a consequential owner boundary.

## Highlights

- Passing verification no longer implies an acceptance verdict for a dependent
  system, shared contract, policy, publication, or external effect.
- Applicable runs record the accountable owner, requested verdict, accepted
  scope, rationale, and wrongness response in `control.md`.
- Missing verdicts remain recoverable through `waiting_human`; no lifecycle
  state or canonical artifact was added.
- Brownfield readiness now surfaces implicit historical, operational,
  compatibility, and ownership constraints before autonomy rises.
- Learning proposals require explicit owner approval before they can mutate
  governance or shared policy.
- Pressure scenarios 070-073 cover acceptance, dependent-system ownership,
  brownfield constraints, and learning promotion. Scenario 024 preserves
  lightweight local work.

## Compatibility

- `control.md` remains current truth and stores conditional acceptance decisions.
- `verification.md` remains evidence truth.
- Existing run records require no migration when no acceptance boundary applies.
- Skill count, runtime adapter model, manifest routing, and lifecycle vocabulary
  remain unchanged.

## Upgrade

Reinstall or update the Kybernetes skills from the repository. Existing durable
records remain valid; add the acceptance section only when a consequential owner
verdict is required.
